# SCAFFOLD PROMPT — Vite + React + TypeScript boilerplate

> **You are an AI coding agent (Claude Code or Codex).** Your job is to scaffold a brand-new
> project in the **current empty directory** by creating every file listed in the
> **FILE MANIFEST** below with its exact contents, then running the **SETUP & VERIFICATION**
> steps until all gates are green. Do not skip files. Do not invent extra files. The route
> tree and lockfile are generated by tooling — see the notes.

## How to use this prompt

1. Create (or `cd` into) an empty directory for the new project.
2. Start Claude Code or Codex there.
3. Paste this whole file (or save it as `SCAFFOLD.md` and say *"follow SCAFFOLD.md"*).
4. Let the agent create the files and run the verification gates.

**Prerequisites:** Node.js ≥ 20 and `pnpm` (`npm install -g pnpm`, or enable Corepack).

## Parameters (substitute before/while creating files)

- `<PROJECT_NAME>` — default **`vite-react-boilerplate`**. Appears in `package.json` (`name`),
  `index.html` (`<title>`, currently "Vite React Boilerplate"), `README.md` (title), and the
  agent guide files. The e2e test asserts the `<title>` — if you change the title, update
  `e2e/example.spec.ts` to match.
- `<AUTHOR>` — default **`Vernon Wee Hong KOH <vernonweehongkoh.developer@gmail.com>`**.
  Appears in `package.json` (`author`) and `LICENSE` (copyright holder).

If the user gives different values, substitute them; otherwise create the files exactly as written.

## Conventions you must honor (the ESLint config is strict — `--max-warnings 0`)

Code you write must satisfy these or `pnpm lint` fails the build:

- **Explicit return types on every function/component.** React components return
  `FunctionComponent` (`React.ReactElement | null`) imported from `@/common/types`.
- **`Array<T>`**, never `T[]` (`@typescript-eslint/array-type: generic`).
- **`import type { ... }`** for type-only imports. Merge value + type imports from the **same**
  module into one statement (`no-duplicate-imports`), e.g. `import { useQuery, type UseQueryResult } from "..."`.
- **JSX props sorted** (`react/jsx-sort-props`): reserved first, shorthand first, callbacks
  last, multiline last. Run `pnpm lint:fix` to auto-sort.
- **No void-returning arrow shorthand** in handlers (`no-confusing-void-expression`) — use a block body.
- **Import via the `@/*` alias** (→ `src/*`), not deep relative paths.
- **Tabs** for indentation (Prettier `useTabs: true`). Run `pnpm format` if unsure.

## Getting Started

Getting started is a simple as cloning the repository

```
git clone <your-repo-url> vite-react-boilerplate
```

Changing into the new directory

```
cd vite-react-boilerplate
```

Removing the .git folder (and any additional files, folders or dependencies you may not need)

```
rm -rf .git
```

Installing dependencies

```
pnpm install
```

And running the setup script (initializes git repository and husky and installs playwright)

```
pnpm run setup
```

Congrats! You're ready to starting working on that new project!

**Note**: This project comes with two git hooks added by [husky](https://typicode.github.io/husky/). A prepare-commit-msg hook to run the [Commitizen](https://github.com/commitizen/cz-cli#readme) cli for those nice commit messages and a commit-msg hook to run [Commitlint](https://commitlint.js.org/#/) on the message itself. Commitlint will ensure the commit message follows the [Conventional Commits specification](https://www.conventionalcommits.org/en/v1.0.0/) (it will if you used commitizen).

If you wish to remove any hooks, simply delete the corresponding file in the .husky directory.

## Important Note

1. This boilerplate does not include a demo. At most, a few utilities (types, devtools wrappers, an initial home route) are included. There is no glue to get in your way when modifying the template.

2. Due to empty directories not being included in git commits, placeholder README files have been added to these empty directories. These README files contain simple descriptions about how the different directories in the accompanying folder structure may be used. As an example check out the [recommended component organizational structure](src/components/README.md) as well as the [recommended folder structure](src/features/README.md).

3. [Faker](https://fakerjs.dev/) is included to encourage more isolated testing and allow for rapid development of demos and MVPs. However, please make note that, [due to a bug](https://github.com/faker-js/faker/issues/1791), importing Faker from the main package (without a locale) will result in the entire Faker lib being imported causing bundle sizes to increase up to 2+ MB. Instead prefer [localized imports](https://fakerjs.dev/guide/localization.html#individual-localized-packages) as shown below.

   ```
   // import { faker } from '@faker-js/faker';
   import { faker } from '@faker-js/faker/locale/en'; // prefer localization when possible
   ```

   The imported lib will instead be around 600 KB. Nonetheless, Faker should **NOT** be used in production and instead be limited to testing and demos.

4. The `@/*` path alias maps to `src/*` (configured in `tsconfig.json` and `vite.config.ts`). Prefer `import { isProduction } from "@/common/utils"` over deep relative paths like `../../../common/utils`.

## Testing

Unit testing is handled by React Testing Library and Vitest while End-to-End (E2E) Testing is conducted by Playwright.

If you'd like to run all tests, Unit and E2E alike, execute the following command:

```
pnpm run test
```

### Unit Testing

When running unit test scripts, it is assumed that unit tests will be colocated with the source files. Take a look at the placeholder README file in `src/components` for [an example](src/components/README.md).

If you'd like to execute unit tests specifically, the below command will execute vitest:

```
pnpm run test:unit
```

If instead you are interested in coverage reporting, run:

```
pnpm run test:unit:coverage
```

### End-to-End (E2E) Testing

Running E2E tests use a similar syntax to running unit tests:

```
pnpm run test:e2e
```

If you wish to see the reports, run:

```
pnpm run test:e2e:report
```

### Devtool dependencies:

- [@tanstack/react-query-devtools](https://tanstack.com/query/v4/docs/react/devtools) - Dedicated dev tools to help visualize the inner workings of React Query
- [@tanstack/router-devtools](https://tanstack.com/router/v1/docs/devtools) - Dedicated dev tools to help visualize the inner workings of TanStack Router

- [@hookform/DevTools](https://react-hook-form.com/dev-tools) - React Hook Form Devtools to help debug forms with validation

A set of utility components are provided in `src/components/utils/development-tools/`. These [wrapper components](https://tanstack.com/router/v1/docs/devtools#only-importing-and-using-devtools-in-developmentgit) check whether the application is running in development or production mode and render the component or null respectively. In other words, you can confidently use them during development without having to worry about them showing up for end users in production.

**TanStack Query Devtools** and **TanStack Router Devtools** are wired up and active in `src/App.tsx`. The Query devtools icon appears in the bottom-left, the Router devtools in the bottom-right.

**TanStack Table Devtools** and **React Hook Form DevTools** wrappers are provided in `src/components/utils/development-tools/` for you to mount inside your own tables/forms. Both require a prop from a specific hook (`table` from `useReactTable()`, `control` from `useForm()`), and must be added to each unique table/form. Example:

```
function Table(): FunctionComponent {
  /* some code */

  const table = useReactTable({
    data,
    columns,
    getCoreRowModel: getCoreRowModel(),
  });

  return (
    <>
      {/* table code */}
      <TanStackTableDevelopmentTools table={table} />
    </>
  )
}
```

import { useTranslation } from "react-i18next";
import type { FunctionComponent } from "@/common/types";

export const Home = (): FunctionComponent => {
	const { t, i18n } = useTranslation();

	const onTranslateButtonClick = async (): Promise<void> => {
		if (i18n.resolvedLanguage === "en") {
			await i18n.changeLanguage("es");
		} else {
			await i18n.changeLanguage("en");
		}
	};

	return (

			<p className="text-white text-6xl">{t("home.greeting")}</p>
			<button
				className="hover:cursor-pointer"
				type="submit"
				onClick={onTranslateButtonClick}
			>
				translate
			</button>
		</div>
	);
};

## Conventions (enforced by ESLint — the build fails on any warning)
- Every function/component needs an explicit return type. React components return `FunctionComponent` (`React.ReactElement | null`) from `@/common/types`.
- Use `Array<T>`, not `T[]`.
- Use `import type { ... }` for type-only imports; merge value + type imports from the same module into one statement (`no-duplicate-imports`).
- JSX props are sorted (`react/jsx-sort-props`): reserved first, shorthand first, callbacks last, multiline last. Run `pnpm lint:fix` to auto-sort.
- Event handlers must not return a value in arrow-shorthand (`no-confusing-void-expression`) — use a block body.
- Import via the `@/*` alias (→ `src/*`), not deep relative paths.
- Indentation is tabs (Prettier `useTabs`).

## Gotchas
- **The route tree is generated.** `src/routeTree.gen.ts` is written by the TanStack Router Vite plugin — never hand-edit it. After adding/renaming a file in `src/routes/`, run `pnpm dev` (or `pnpm exec vite build`) to regenerate it. On a fresh clone it does not exist yet, so generate it before `tsc` / `pnpm build`.
- **i18n keys are typed.** `t()` keys are derived from `src/assets/locales/en/translations.json`. Add new keys there (and mirror them in `es/`) or TypeScript rejects them.
- **Devtools are dev-only.** Use the wrappers in `src/components/utils/development-tools/` (they render `null` in production). Router + Query devtools are mounted in `src/App.tsx`; Table/RHF devtools mount inside the specific table/form via the wrappers.
- **pnpm build scripts.** `esbuild` and `@swc/core` need their native postinstall; they are approved in `pnpm-workspace.yaml` (`allowBuilds`). If a pnpm prompt appears, run `pnpm rebuild esbuild @swc/core`.

# SETUP & VERIFICATION

Run these in order. Treat steps 5–7 as **hard gates**: fix any drift before reporting done.

1. **Init git** (husky needs a repository):
   ```
   git init
   ```
2. **Create all files** from the manifest above. Make the husky hooks executable:
   ```
   chmod +x .husky/commit-msg .husky/prepare-commit-msg
   ```
3. **Install dependencies** (also runs the `prepare: husky` hook):
   ```
   pnpm install
   ```
   `esbuild` and `@swc/core` build scripts are pre-approved in `pnpm-workspace.yaml`. If your
   pnpm version still reports ignored builds, run `pnpm rebuild esbuild @swc/core`.
   (No pnpm? `npm install -g pnpm`, or `corepack enable`.)
4. **Generate the route tree before typechecking.** `src/main.tsx` imports `@/routeTree.gen`,
   which doesn't exist yet, and `pnpm build` runs `tsc` first. Generate it with one Vite pass
   (Vite transpiles without typechecking, so it succeeds and writes `src/routeTree.gen.ts`):
   ```
   pnpm exec vite build
   ```
   (Alternatively start `pnpm dev` and stop it once it prints "ready".)
5. **Lint — must be 0 warnings** (`eslint --max-warnings 0`):
   ```
   pnpm lint
   ```
6. **Typecheck — must be clean:**
   ```
   pnpm exec tsc -p tsconfig.json --noEmit
   ```
7. **Build — `tsc && vite build` must succeed:**
   ```
   pnpm build
   ```
   `dist/` should contain `index.html`, `assets/`, and a copied `locales/` folder.
8. **(Optional) E2E** — the Playwright config auto-boots `pnpm dev` on port 5173:
   ```
   npx playwright install
   pnpm test:e2e
   ```
   Note: Playwright may not download browsers on very new/unsupported OS versions.
